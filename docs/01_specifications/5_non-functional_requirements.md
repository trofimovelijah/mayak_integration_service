## 3.4 Нефункциональные требования

| ID требования | Требование | Описание |
| ----- | ----- | ----- |
| Требования к производительности |  |  |
| НФТ-1.1 | Время обработки триггера | Время от получения триггера до отправки API запроса во внешний сервис не должно превышать 5 секунд |
| НФТ-1.2 | Время отправки email | Благодарственные письма за пожертвования должны отправляться не позже чем через 2 минуты (или 30 секунд?) |
| НФТ-1.3 | Пропускная способность | Платформа должна обрабатывать до 1000 триггеров в час в пиковые нагрузки |
| НФТ-1.4 | Время отклика | Время отклика административного интерфейса \- не более 3 секунд |
| НФТ-1.5 | Производительность отправки | Платформа должна обеспечивать отправку до 100 писем в день на начальном этапе в месяц |
| Требования к надёжности |  |  |
| НФТ-2.1 | Доступность системы | Платформа должна быть доступна 99.5% (не более 3.65 часов простоя в месяц) в рабочее время (9:00-18:00 по МСК) |
| НФТ-2.2 | Восстановление системы | Данные о конфигурации и логах должны резервироваться ежедневно с возможностью восстановления за 4 часа |
| НФТ-2.3 | Сохранность данных при сбоях | Система должна обеспечивать сохранность данных при сбоях (zero data loss) |
| НФТ-2.4 | Повторная отправка  | Механизм повторных попыток отправки при временной недоступности внешних сервисов |
| НФТ-2.5 | Устойчивость к сбоям API | При недоступности внешних API система должна сохранять задачи в очереди и повторять попытки |
| НФТ-2.6 | Допустимость ошибок | Платформа должна гарантировать не более 0,5% ошибок доставки писем в год |
| НФТ-2.7 | Уровень обслуживания  | Платформа должна поддерживать 99,5% успешных отправок писем в год |
| НФТ-2.8 | Действия при отказоустойчивости | При сбое в одном из модулей система должна продолжать работу для других каналов и сохранять данные о сбое для последующего восстановления. |
| НФТ-2.9 | Рост нагрузки | Платформа должна выдерживать 5-кратное увеличение нагрузки (до 5000 триггеров в час) |
| Требования к безопасности |  |  |
| НФТ-3.1 | Соблюдение 152-ФЗ | Полное соблюдение требований ФЗ-152 о защите персональных данных без трансграничной передачи |
| НФТ-3.2 | Аутентификация API | Все API запросы должны использовать безопасную аутентификацию (API ключи, токены JWT) |
| НФТ-3.3 | Шифрование данных | Все персональные данные должны храниться в зашифрованном виде (AES-256) |
| НФТ-3.4 | Ограниченный доступ к админ-панели | Аутентификация и авторизация для доступа к административному интерфейсу |
| НФТ-3.5 | Аудит данных | Ведение журнала аудита операций с персональными данными |
| Требования к совместимости |  |  |
| НФТ-4.1 | API совместимость Unisender Go | Совместимость с API сервиса рассылок Unisender Go |
| НФТ-4.2 | API совместимость CloudPayment | Совместимость с платёжной системой CloudPayment |
| НФТ-4.3 | API совместимость MixPlat | Совместимость с платёжной системой MixPlat |
| НФТ-4.4 | Совместимость Битрикс | Совместимость с БД платформы "Давайте соберемся" (Битрикс) |
| Требования к сопровождаемости |  |  |
| НФТ-5.1 | Логирование событий | Все события системы должны логироваться с timestamp, уровнем важности и контекстом |
| НФТ-5.2 | Мониторинг здоровья интеграционных точек | В ходе сопровождения Платформы необходимо осуществлять мониторинг инфраструктуры, включая все интеграционные взаимодействия |
| НФТ-5.3 | Выборочное отключение уведомлений | Возможность отключения отдельных типов уведомлений без остановки системы |
| НФТ-5.5 | Документированные процедуры резервного копирования и восстановления | Перед внедрением Платформы должен быть разработан регламент резервного копирования и восстановления |
| Требования к масштабируемости |  |  |
| НФТ-6.1 | Длительное хранение | Хранение истории коммуникаций за последние 3 года |
| НФТ-6.2 | Требование к размеру базы данных | Поддержка базы данных до 10 000 контактов |
| НФТ-6.3 | Горизонтальное масштабирование | Платформа должна поддерживать добавление новых экземпляров для увеличения производительности |
| НФТ-6.4 | Архивация устаревших данных | Архивация устаревших данных с сохранением статистики |
